# NoonFlow 使用说明

NoonFlow 是面向 `Claude Code` 与 `Codex` 的桌面 GUI，用于在图形界面中管理项目会话、运行时配置、Git worktree 以及相关的历史、变更和成本信息。

本文档说明安装方式、配置规则和主要功能入口。

---

## 1. 安装

安装步骤：

1. 双击 `.dmg`
2. 将 `NoonFlow.app` 拖入"应用程序"
3. 启动应用

### macOS 安全提示

如果系统阻止启动，请按以下两种处理方式：

1. 前往 `系统设置 -> 隐私与安全性`，点击"仍要打开"
2. 在"应用程序"中找到 `NoonFlow`，右键选择"打开"

必要时也可执行：

```bash
xattr -cr /Applications/NoonFlow.app
```

---

## 2. 首次启动前提

NoonFlow 使用过程中会依赖以下本地组件：

- Node.js
- Git
- Claude Code CLI
- Codex CLI

首次启动后，建议先进入 `Settings` 检查运行时状态，如果本机缺少依赖，优先使用设置页中的一键安装功能。命令行安装仅作为备用方案。

### 命令行备用安装方式

```bash
# Claude Code
npm install -g @anthropic-ai/claude-code
claude login

# Codex
npm install -g @openai/codex
codex login
```

---

## 3. Claude Code 配置注意事项

如果本地曾修改过 Claude Code 的配置，尤其是以下文件：

- `~/.claude/settings.json`
- `~/.claude.json`

或者使用过类似 `cc switch` 之类会改写 Claude Provider 配置的工具，建议先恢复默认配置，或直接在 NoonFlow 中重新配置。

原因是 NoonFlow 会读取本地 Claude 相关配置；如果这些配置经过较多自定义修改，可能导致运行时行为异常或 Provider 识别不一致。

---

## 4. Provider 配置规则

NoonFlow 中的 `Provider` 主要用于第三方服务商、第三方转发链路或自定义 API 配置。

并不是所有场景都需要额外配置 Provider。

| 场景 | 是否需要配置 Provider | 说明 |
|------|----------------------|------|
| 本地 `Claude Code` 已完成登录 | 否 | 可直接使用默认链路 |
| 本地 `Codex` 已完成登录 | 否 | 可直接使用默认链路 |
| 使用第三方服务商 | 是 | 需要配置 API Key / Base URL |
| 使用自定义 API Key 或自定义 Base URL | 是 | 需要在 `Settings` 中配置 |

建议首次使用时只保证一条最小可用链路：

- `Claude Code` 可用
- 或 `Codex` 可用

不建议一开始同时配置多套 Provider、多组模型和复杂的自定义 Base URL。

## 5. 功能概览

### 5.1 Workspace

NoonFlow 的核心能力是将 `Claude Code` 与 `Codex` 的命令行工作流放入图形界面中。

在 `Workspace Chat` 中，可以围绕具体项目目录完成以下操作：

- 在同一 `Workspace` 下，于同一窗口中管理多个会话 `Tab`
- 为不同 `Tab` 选择不同的运行时、模型和服务商
- 在同一个工作区内同时使用 `Claude Code` 与 `Codex`
- 在 `Codex` 路径下按需切换不同模型、不同厂商或不同 Provider
- 在统一界面中结合项目目录、文件、会话状态与历史信息持续工作

这是整个产品的主入口，也是最核心的使用场景。

### 5.2 Worktree 与多工作区

NoonFlow 支持 Git worktree 工作流，可用于：

- 在多个分支目录之间切换
- 为不同工作目录维护独立会话
- 在多工作区之间组织不同任务和上下文

这部分能力主要解决"命令行能做，但不够直观、不够好管理"的问题。

### 5.3 监控与可视化

NoonFlow 提供了若干可视化页面，用于观察当前使用情况，而不仅仅是停留在聊天窗口中。

相关能力包括：

- 成本与用量统计
- 工具调用与执行情况可视化
- 会话与活动情况的聚合展示
- 运行过程中的历史与状态回看

### 5.4 Session 监控

围绕 Session，NoonFlow 提供了更适合图形界面的观察能力，例如：

- 会话列表和切换
- 会话历史查看
- 会话状态与活动记录
- 会话相关变更的回看

这部分能力适合用于管理长期项目和多任务并行场景。

### 5.5 Memory 与时间线回放

除实时聊天外，NoonFlow 也强调对历史信息的回看能力。

相关能力包括：

- Memory 入口
- 历史会话检索
- 时间线式回放
- 对过去上下文和过程的重新查看

这部分能力更偏向"回顾、检索、追踪"，而不是单次问答。

### 5.6 Skills、Agents 与 Hooks 的可视化管理

除聊天本身外，NoonFlow 也提供了对运行时相关配置的可视化管理能力。

这部分主要包括：

- `Skills`：查看和管理可复用技能
- `Agents`：查看和管理角色定义
- `Hooks`：查看和管理运行时钩子

这类能力的重点不是"增加一个新概念"，而是把原本偏命令行、偏文件配置的内容，用更直观的界面方式组织和管理。

### 5.7 MCP

`MCP` 相关能力已经纳入产品范围，但考虑到当前实际使用场景相对较少，这部分功能目前暂时隐藏，未作为默认主入口展示。

### 5.8 通用设置

`Settings` 中有几项通用开关会直接影响日常使用体验：

1. `自动批准所有操作`
用于控制是否跳过权限确认。开启后，运行时执行工具操作时不再逐项确认，适合非常明确且可信的任务场景；首次使用通常不建议开启。

2. `显示思考过程`
用于控制是否展示模型的 thinking 内容。开启后，可以在聊天中看到折叠的思考过程；代价通常是响应更长、消耗更多。

3. `Show Widget / 生成式 UI`
用于控制是否允许在回复中生成可视化 widget，例如图表或结构化展示内容。适合需要更强可视化表达的场景。

4. `上下文使用进度条`
用于显示当前会话的上下文占用情况，便于观察上下文窗口消耗和会话长度变化。

对于首次体验用户，建议优先关注 `Workspace Chat`、监控可视化、Session 观察、Memory / 时间线回放，以及 `Skills / Agents / Hooks` 的可视化管理。

## 6. 常见说明

### 6.1 应用已启动，但运行时不可用

请优先检查：

- `Settings` 中所需运行时是否显示可用
- 本地 CLI 是否已安装
- 本地 CLI 是否已完成登录
- Provider 或模型配置是否有效

### 6.2 请求持续失败

常见原因包括：

- API Key 无效
- 额度不足
- Base URL 配置错误
- 网络受限
- 模型与服务商不匹配

### 6.3 CLI 已安装，但应用无法正常识别

这通常与以下因素有关：

- 本地命令可执行，但当前应用环境未正确识别
- 本地 CLI 尚未登录
- Claude 本地配置被修改，导致读取异常
- 应用重启前状态尚未刷新

## 7. 本地数据位置

NoonFlow 默认将本地数据保存在：

```bash
~/.noonflow/
```

核心数据库文件为：

```bash
~/.noonflow/noonflow.db
```

该目录通常包含会话数据、运行时配置和相关本地状态信息。
